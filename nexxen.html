Nexxen DSP: Rebuilding a Foundation for Scale

A programmatic advertising platform with no design system. Engineers hard-coding values, designers duplicating work, product surfaces drifting apart. I rebuilt the system from primitives up so the team could ship consistently and fast.

Role

Product Designer (systems lead)

Company

Nexxen

Platform

Web application (Figma)

Timeline

6 months

200+ Components rebuilt
40+ Hard-coded values eliminated
7 Color palettes supported
1 Source of truth for all themes

The problem with organic growth

Nexxen's DSP had grown fast. The product surface had expanded, the team had grown, and the design had kept up by adding. New components when something was needed. New colors when branding evolved. Hard-coded values because there was no other option.

By the time I joined, the result was a system in name only. Buttons that looked similar but weren't the same component. Color values that appeared in 40+ places with no single source. Dark mode had been requested for two product cycles but no one knew how to implement it without touching every screen by hand. White-label retheming for enterprise clients was theoretically on the roadmap and practically impossible.

The system was not broken. It was just not a system. It was a collection of past decisions with no organizing logic underneath them.

"The question was never whether to fix it. The question was whether to patch the existing components or start from the token layer. Patching would have been faster. Starting from tokens was the only way to make the problem go away."

Audit before architecture

Before proposing anything, I audited the existing system: every component, every color value, every spacing instance across the three main product surfaces. The goal was to understand the actual scope before committing to an approach.

The audit surfaced 40+ unique hard-coded color values where there should have been fewer than 15 semantic tokens. Spacing was applied inline with no shared scale. Typography had six variations of the same body style. Components that looked identical to users were built differently in Figma depending on who made them.

[ Audit output: color value inventory and duplication map ]

The audit also revealed something useful: despite the inconsistency, about 60% of the visual decisions were actually correct. The existing color choices, type scale, and spacing rhythm were solid. The problem was not the decisions. It was the lack of structure that made them impossible to maintain.

The token architecture

I designed a 3-tier token structure in Figma using Figma Variables, aligned with the engineering team's token naming conventions so design and code could stay in sync.

Tier 1 — Global tokens

The raw palette. Every literal color, every spacing value, every type size. These are never used directly in components. They exist only to feed the semantic layer. Changing a global token cascades everywhere.

Tier 2 — Semantic tokens

Named by intent, not by value. color.background.primary rather than #FFFFFF. This is the layer that makes theming possible. Swap the global values that semantic tokens point to and the entire product retheems without touching components.

Tier 3 — Component tokens

Scoped to specific components where semantic tokens need further specificity. A button's hover state background is different from a card's hover state background, even if both use the same semantic family. Component tokens hold that nuance without breaking the cascade.

[ Token architecture diagram: global to semantic to component cascade ]

Rebuilding the component library

With the token architecture defined and validated with engineering, I rebuilt the component library from the ground up. The principle was: every component consumes tokens, never raw values. If a component needed a value that wasn't in the token system, that was a signal to add a token, not to hard-code.

I prioritized rebuilding in order of usage frequency: form elements and data tables first (they appear on almost every screen in a DSP), then navigation and layout components, then specialty components. Each component was built with all states covered: default, hover, focused, disabled, error, and loading where applicable.

I also wrote documentation inline in Figma for each component: when to use it, when not to, and what it maps to in the engineering component library. This was not a Confluence document. It was part of the component itself, visible when any designer opened it.

Dark mode and white-label in practice

Once the token architecture was in place, dark mode was not a project. It was a two-hour task. I created a dark mode global token set, mapped it to the existing semantic tokens, and every component switched without a single component-level change. That moment was the clearest proof that the architecture had worked.

White-label retheming for Nexxen's enterprise clients followed the same pattern. A client's brand palette mapped into a global token set, and their theme cascaded through the entire product automatically. What had been scoped as a multi-week engineering effort became a configuration task.

How I worked with the team

The most important collaboration in this project was with engineering. Design systems only work if engineers use them, and they only use them if the system maps to how they build. I spent the first two weeks aligning the Figma token naming conventions with the engineering team's existing token structure so that design and code were speaking the same language.

I ran onboarding sessions with the other designers on the team once the core component set was stable. The goal was not to teach them a tool. It was to explain the mental model: why the architecture was structured the way it was, so they could extend it correctly rather than work around it. I also set up a shared Slack channel for component requests and questions so the system could evolve without me being the only person who could touch it.

By the end of the project, two other designers had contributed new components that followed the token structure correctly without my review. That was the real measure of success.

Outcome

The rebuilt design system eliminated the bottlenecks that had slowed the team for two product cycles. Dark mode shipped. White-label retheming became a configuration task. New product surfaces could be built from existing components rather than rebuilt from scratch.

The less visible outcome was the change in how the team worked. Designers stopped making one-off decisions about color and spacing. Engineers stopped receiving specs with hard-coded values. Handoffs became faster because the shared language was already there.

What I'd do differently

I would instrument the system earlier. We had qualitative signals that things were faster, but I didn't have a baseline measurement of how long specific design tasks took before the rebuild versus after. Measuring component reuse rate and time-to-spec for new surfaces would have made the impact concrete instead of felt.

I'd also involve a second designer in the component documentation from the start. Documentation written by the person who built the system is often too close to the reasoning to be useful for someone new. A second perspective during the writing would have caught the gaps.
